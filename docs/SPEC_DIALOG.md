# 対話仕様書作成ガイド (Phase 7.8.2)

`uaf create` は自然言語のリクエストからプロジェクトを生成する前に、**仕様書 (spec.md) を対話で組み立てる**フェーズを挟む。このドキュメントはその仕組みと使い方をまとめたもの。

## なぜ対話なのか

「避けゲーを作って」の一言から生成されるプロジェクトには、以下のような曖昧さが残る:

- 視点（トップダウン / サイドビュー / 一人称）
- 操作（キーボード / マウス / タッチ）
- 勝利条件（エンドレス / スコア / タイム）
- 難易度カーブ / アセット要件 / …

Phase 7.8 以前はこれらを Director エージェントが LLM の推測で埋めていたため、ユーザーの期待と食い違うことがあった。Phase 7.8 は **interviewer エージェントが 3〜7 個の質問で合意を取る** ことで、以降の Architect / Programmer が迷わず実装に入れるようにする。

## 典型的な流れ

```
$ uaf create "避けゲー" --recipe 2d-game

=== spec phase ===
request: 避けゲー

Q1. 視点・ジャンルを教えてください。
  ❯ トップダウン（上から見下ろし）
    サイドビュー
    縦スクロール
    自由入力 (free text)

Q2. プレイヤーの操作方法は？
  …

(5〜7 問ほど続く)

--- spec.md preview ---
# 仕様書: シンプル避けゲー

## コンセプト
トップダウン視点の 2D 避けゲー…

## 機能要件 (MUST)
- …

## 仕様確定の根拠 (Q&A 履歴)
…
--- end of preview ---

この仕様で進めて良いですか?
  ❯ はい — この仕様で実装に進む
    いいえ — 中断する
    編集する — エディタで spec.md を修正
```

## 対話のルール（interviewer prompt 内蔵）

- **質問は 1 回 1 問**（並列実行禁止）。`ask_user` ツールの process-wide mutex でも防衛する
- **3〜7 個で完結**。ハードリミット 12 問（R4 サーキットブレーカー）
- **必ず選択肢を提示**（3〜4 個 + 自由入力オプション）
- **リクエスト本文で既に明らかな項目は聞かない**（例: 「縦スクロールシューティング」と書いてあるなら視点は質問しない）
- **回答はユーザーの言語で行う**（日本語で書いたら日本語で問われ、日本語で書かれる）

## レシピ別の質問テンプレート

`agents/interviewer/prompt.md` に **7 レシピ種別すべての最小質問セット** が内包されている:

| レシピ | 主な質問 |
|--------|---------|
| 2d-game | 視点 / 操作 / 終了条件 / 難易度 / アセット要件 |
| 3d-game | 視点 / 操作デバイス / 物理 / シーン規模 / 終了条件 |
| web-app | ユーザー像 / 認証 / 永続化 / レスポンシブ / デザイン方向 |
| mobile-app | 対象 OS / 主要画面 / オフライン / 同期 / 通知 |
| desktop-app | OS / ファイル関連付け / トレイ常駐 / 操作モード |
| cli | サブコマンド / 入出力形式 / 設定ファイル / 対話モード |
| api | エンドポイント / 認証 / DB / レスポンス形式 / レート制限 |

## 承認フロー

spec.md のプレビュー表示後に 3 択:

| 選択 | 挙動 |
|------|------|
| はい (`y`, デフォルト) | そのまま roadmap phase へ |
| いいえ (`n`) | `USER_ABORT` で終了（exit 8）。再実行して別の答えを入れればよい |
| 編集する (`e`) | さらに 2 択（下記） |

`e` を選ぶと修正方法のサブメニューが出る（Phase 7.8.9 追加）:

| サブ選択 | 挙動 |
|----------|------|
| 対話で修正（自由に指示） | interviewer を **REVISE モード** で再起動。自由記述で「敵を 3 種類に」「難易度を緩やかに」等を伝えると、LLM が該当セクションだけを書き換える（未指定箇所は一字一句保持） |
| エディタで直接編集 | `$EDITOR` で spec.md を開く（従来の挙動） |

修正後は再プレビューされ、再び y/N/e が出る。納得するまで何度でも繰り返せる。

- **上限**: 対話修正は **最大 5 回**（R4 サーキットブレーカー）。6 回目の `e` は自動でエディタ直行になる
- **曖昧な指示**: interviewer が ambiguous と判断した場合、`ask_user` で 1 回だけ確認してから修正する（確認上限 2 回）
- **スタック変更の拒否**: recipe のスタック（Phaser → Three.js など）は REVISE モードでも変更できない。interviewer が代替案を提案する

`--yes` / `-y` フラグで承認プロンプトを完全にスキップできる（CI / スクリプト用）。対話修正も同時にスキップされる。

### 対話修正モードの UX 例

```text

## 機能要件 (MUST)
- スコアアタック
- プレイヤー HP: 3
- 難易度: 固定
...
--- end of preview ---

この仕様で進めて良いですか?
  はい — この仕様で実装に進む

修正方法を選択してください
```

## 非対話モード: `--spec-file`

事前に用意した `spec.md` を使う場合:

```bash
uaf create --spec-file ./my-spec.md --recipe 2d-game
```

- interviewer フェーズをスキップ
- spec.md を workspace にコピーしてそのまま roadmap phase へ
- `userApproved: true` として state.json に記録される
- CI / スクリプト / 過去の spec.md の再利用に使える

## 対話を完全に止めたい（legacy フロー）

Phase 7.8 以前の挙動（Director が spec.md を生成する直接フロー）に戻したい場合:

```bash
uaf create "リクエスト" --no-spec
```

- interviewer / roadmap-builder / 承認プロンプトをすべてスキップ
- 昔の `runOrchestrator` 直呼び出しと 100% 同じ挙動
- state.json には `phase` / `roadmap` が記録されない（legacy 扱い、`uaf resume` 不可）

## 生成される spec.md の構造

```markdown
# 仕様書: <プロジェクト名>

## コンセプト
<1〜3 文>

## 機能要件 (MUST)
- ...

## 機能要件 (NICE TO HAVE)
- ...

## 非機能要件
- パフォーマンス / アクセシビリティ / セキュリティ

## UI / フロー
<mermaid 図 or 箇条書き>

## 技術ヒント (optional)
<ユーザーの希望。recipe のスタックは上書きしない>

## 仕様確定の根拠 (Q&A 履歴)
- Q: … / A: …
```

最後の **Q&A 履歴** セクションは、後から「どの前提で作ったか」を追える監査ログの役割を持つ。

## 対話 UX のチューニング

- 質問が多すぎると感じたら: `agents/interviewer/prompt.md` の「質問回数は最大 7 回」を減らす
- 質問テンプレを変えたい: 同ファイル内のレシピ別セクションを編集
- 質問の言い回しは LLM に任されている。テンプレはガイドであり、ユーザーのリクエスト本文によって interviewer が柔軟に変える

## 既知の制約

- 実装段階に入ってから「やっぱり仕様を変更したい」は現状 iterate コマンド経由（またはゼロからやり直し）でしか対応できない
- spec.md の途中編集 (e オプション / 対話修正) 後に roadmap を再生成する機能は未実装。roadmap は最終承認された spec に合わせて作られる
- 対話修正 (`e` → 1) 1 回あたり interviewer に 1 度 LLM コールが走る。Sonnet 4.6 で概ね $0.01〜$0.05 程度（spec.md の分量と修正範囲に依存）
